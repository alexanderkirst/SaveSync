GBAsync — Plan: Quick wins (v0.1.6 UX roadmap)
================================================
Status: **shipped in v0.1.6** (Switch/3DS clients, Mar 2026). This file kept as design reference.

Source backlog: SaveSync/docs/TODO.md — “### Quick wins”

Scope: three items — sync preview, status line, per-device save lock.
  (Server data paths are already configurable via repo-root `.env`: `SAVE_ROOT`, `INDEX_PATH`,
  `HISTORY_ROOT` — see `server/README.md` “Data directory”.)
Code anchors: switch-client/source/main.cpp (`run_sync`, `preview_auto_plan`, status helpers);
  3ds-client/source/main.c (`run_sync`, `preview_auto_plan_3ds`, status helpers).


1) Sync preview screen before apply
-----------------------------------

Goal
  After the user chooses **Auto (full sync)**, run the same discovery work you already need
  (local scan, GET /saves, baseline load) and compute **what would change** — without writing
  saves or updating the baseline yet. Then show a **preview** screen listing per game_id
  whether the run would: upload, download, skip (with reason), or stop for **conflict**
  (needs X/Y/B later). Only after the user **confirms on that preview** does the app apply
  the plan (PUT/GET, conflict prompts, baseline writes).

  In short: **Auto → scan + plan → preview / confirm → apply.** The old standalone “confirm
  full sync” screen goes away; **this preview is the confirmation** (you are confirming the
  concrete plan, not a generic warning).

UX flow (both platforms)
  1. Main menu → user selects **full sync (Auto)**.
  2. App performs **read-only** work: list locals, fetch remote metadata, load baseline —
     same inputs as today’s merge.
  3. App runs **pure planning** (no disk writes except whatever reads you already do for
     hashing; no PUT/GET to transfer save bodies if you can decide from hashes alone — you
     already have local SHA and remote SHA from /saves).
  4. **Preview screen** shows counts and per-id rows (upload / download / skip / conflict).
  5. User confirms (**A** / equivalent) → **apply** phase runs existing logic (including
     interactive conflict resolution where needed). **B** / back → return to main menu with
     no changes.

Why the code change is non-trivial
  Today Auto interleaves decisions and I/O in one pass (Switch: run_sync() ~L1135+;
  3DS: run_sync AUTO branch ~L1713+). Conflict resolution already blocks in
  resolve_both_changed_conflict_*() mid-loop — that stays in **apply**, after preview confirm.

Recommended approach
  A) Extract a pure “plan” phase (no writes, no network transfer of blobs beyond existing
     GET /saves metadata):
     - Inputs: same as today — local scan + remote list + baseline.
     - For each id in the merge set, compute an enum/classification:
         OK (hashes match), UPLOAD, DOWNLOAD, SKIP_NO_BASELINE, SKIP_BOTH_CHANGED,
         CONFLICT_BOTH_CHANGED, etc.
     - For CONFLICT_BOTH_CHANGED, the plan row should say “needs interactive resolution”;
       X/Y/B runs only in the apply phase after the user accepted the preview.

  B) **Apply** pass: after preview confirm, reuse existing put/get/conflict helpers in the
     same order as today so behavior stays identical.

  C) UI (Switch): remove confirm_auto_sync(); after choose_action selects Auto, run plan →
     preview UI → on success proceed to apply. Paginate or truncate long lists if needed.

  D) UI (3DS): same flow; replace confirm_auto_sync() with preview after Auto.

Edge cases to specify in implementation
  - Upload-only / Download-only: optional preview in a later iteration; this item is **Auto** first.
  - Large game lists: scroll, page, or “+ N more” on constrained consoles.

Rebuild note (repo rule)
  After C++ changes under switch-client/ or 3ds-client/, run from repo root:
  ./scripts/release-switch.sh dev
  ./scripts/release-3ds.sh dev


2) Last sync health / status line
---------------------------------

Goal
  On the main menu, show a compact line (or two) for:
  - Last successful sync (time or “never”),
  - Whether the server was reachable on last attempt,
  - Last Dropbox sync-once result (if user triggered it or you decide to probe).

Implementation sketch
  - Persist a tiny state file next to the baseline, e.g.
    sdmc:/.../.gbasync-status (JSON or key=value) — same directory as
    active save dir so VC vs normal mode stay consistent on 3DS (active_save_dir()).

  Fields (example)
    last_sync_utc / last_sync_epoch
    last_sync_ok (bool)
    last_server_reachable (bool) — set from GET /saves or GET /health outcome
    last_dropbox_sync_ok (bool or tri-state unknown)
    last_error_short (optional, capped string)

  Update points
    - After any sync path completes (success or failure): write atomically (write temp + rename).

  UI
    - Switch: under “Server:” / “Save dir:” on main menu (~L1298+).
    - 3DS: same region (~L1900+).

  Optional: lightweight probe on menu idle — avoid hammering server.


3) Lock save per device (prevent overwrite / push)
-------------------------------------------------

Goal
  Mark specific game_ids as “locked” on this device so Auto (and optionally manual modes)
  will not upload, download, or overwrite local file for those ids.

Design options (pick one for v1)

  A) Client-only lock (simpler, matches “per device”)
     - New config list, e.g. [locks] or sync.locked_ids=comma-separated in config.ini.
     - In Auto loop: if id is locked, SKIP with log “locked on this device”.
     - Manual Upload/Download: either respect lock or allow override with extra confirm —
       decide explicitly.

  B) Server-enforced lock (stronger)
     - Requires metadata in index or separate store + API to set/clear locks per key.
     - PUT could reject with 423 or 409; clients must handle.
     - Higher effort; couples with admin panel (longer-term).

Suggested v1
  - Implement A) on Switch + 3DS for Auto; document format.
  - Store locks in config, not baseline file, so baselines stay hash-only.

Testing
  - Locked id with divergent local/server: must not pull/push; user should see SKIP lines.


Cross-cutting: ordering of work
-------------------------------
Suggested sequence to reduce rework:
  1) Status file + main-menu status line (isolated).
  2) Config: add locks parsing + merge behavior (item 3).
  3) Refactor merge into plan + apply; add preview UI after Auto (item 1); remove old confirm for Auto.
  4) Release notes + USER_GUIDE updates.
